# Infra Provisioning

This package owns the shared platform stack for staging in `packages/infra`.

Today that means:

- Azure resource group
- AKS cluster
- Azure Container Registry
- Shared Postgres server
- Flux bootstrap that syncs `packages/infra/manifests/clusters/staging` from the
  remote `ci` branch

It does not fully provision a working app environment by itself. There are still
manual or app-specific steps after `pulumi up`.

## Prerequisites

Install these CLI tools before running any commands:

- **Azure CLI**
  ([install](https://learn.microsoft.com/en-us/cli/azure/install-azure-cli)) —
  needed for `az login` and SP/role management.
- **Pulumi CLI** ([install](https://www.pulumi.com/docs/install/)) — needed for
  `pulumi login` and `pulumi up`.
- **Node.js** — match the version in `.node-version` or `.nvmrc` at the repo
  root.
- **pnpm** — the monorepo package manager. If you use `corepack enable`, pnpm is
  available automatically.
- **mise** ([install](https://mise.jdx.dev/getting-started.html)) — manages env
  vars from `.mise.local.toml` generated by bootstrap. The bootstrap script
  creates this file but cannot activate the shell for you.
- **Docker** (optional) — only needed when building and pushing app images
  locally (`docker buildx build --push`).

## Fresh staging bootstrap

Run these commands from `packages/infra`.

1. Install repo dependencies with `pnpm install` from the workspace root.
2. Log into Azure with a privileged identity: `az login` The identity needs
   **Owner** (or equivalent) on the target subscription so the bootstrap script
   can create a service principal and assign roles.
3. Log into Pulumi: `pulumi login` This authenticates against your Pulumi
   backend (cloud or self-hosted). Skip this if you previously ran
   `pulumi login` and the session is still valid.
4. Run `pnpm bootstrap:staging`.
5. Trust the generated mise env file and reload your shell:
   ```bash
   mise trust <repo-root>/.mise.local.toml
   mise env
   ```
   Or close and reopen your terminal in the repo root. Bootstrap creates two
   files:
   - `packages/infra/.env.staging.local` — Azure auth env vars (ARM/AZURE client
     ID, secret, subscription, tenant). Gitignored.
   - `<repo-root>/.mise.local.toml` — same env vars in mise format, auto-loaded
     when you enter the repo directory with mise active.
6. Run `pulumi up --stack staging`.

`pnpm bootstrap:staging` does three important things that are easy to miss on a
fresh machine:

- Creates or reuses the `aamini-platform-staging` Azure service principal.
- Grants the Azure roles required for Pulumi and AKS access.
- Writes the local Azure auth env vars and stores these Pulumi secrets for the
  `staging` stack:
  - `aamini-platform:cloudflareApiToken`
  - `aamini-platform:githubApiToken`
  - `aamini-platform:dbPassword`

If `pulumi up` fails immediately on Azure auth or missing secret config, rerun
`pnpm bootstrap:staging` first.

## Drift recovery

If Azure resources were deleted out of band, the Pulumi stack can still contain
stale state. In that case `pulumi up --stack staging` may fail with errors like
`ResourceGroupNotFound`.

Recover with:

```bash
pnpm refresh:staging
pulumi up --stack staging
```

Use this before trying to recreate the platform after a resource group, AKS
cluster, or Postgres server was manually deleted.

## What `pulumi up` does not cover

### Cloudflare DNS is now tied to the platform ingress IP

The platform stack now reserves a static Azure public IP for Traefik and uses
that same IP for Pulumi-managed Cloudflare DNS.

Current implication:

- `pulumi up --stack staging` now owns the wildcard DNS target again.
- Cluster rebuilds keep the same public IP instead of forcing a manual
  Cloudflare update.
- The Cloudflare provider authenticates from the existing
  `aamini-platform:cloudflareApiToken` stack secret.

### Flux tracks remote Git, not local files

Flux is configured in `src/aks.ts` to sync from:

- repo: `https://github.com/aamini-stack/projects`
- ref: `refs/heads/ci`
- path: `packages/infra/manifests/clusters/staging`

Current implication:

- Local edits in this workspace do not reach the cluster by themselves.
- Manifest changes must be pushed to `ci` before Flux will reconcile them.
- For emergency recovery, you may need to `kubectl apply` the updated manifest
  directly, then commit and push the same change afterward.

### App infra is separate

`packages/infra` only creates the shared platform resources. App-specific
database users, databases, and app-level Pulumi resources still live in each app
package.

For `imdbgraph`, continue in `apps/imdbgraph/infra` after the platform stack is
healthy.

## Fresh cluster follow-up checklist

After a brand new cluster comes up, these are the remaining manual steps for
`imdbgraph`.

1. Refresh kubeconfig with
   `az aks get-credentials --resource-group rg-aamini-staging --name aks-cluster-staging --overwrite-existing`.
2. Run the app stack in `apps/imdbgraph/infra`.
3. Reseal any checked-in sealed secrets against the current cluster cert.
4. Build and push the app image.
5. If Flux is not reconciling from Git successfully, apply the local manifests
   directly as a recovery step.
6. Populate the staging database.
7. Verify Flux/Traefik/cert-manager/app health.
8. Run `pnpm e2e:staging` in `apps/imdbgraph`.

## Resealing secrets after cluster rebuild

Checked-in sealed secrets stop working after sealed-secrets key rotation or
cluster recreation.

Symptom:

- The `SealedSecret` still exists in Git.
- The runtime `Secret` is never created in the new cluster.

Recovery workflow:

1. Update `apps/imdbgraph/.env.local` with the plaintext runtime values.
2. Run
   `node --experimental-strip-types ../../scripts/src/aamini.ts secrets seal imdbgraph`
   from `apps/imdbgraph`, or
   `node --experimental-strip-types ./scripts/src/aamini.ts secrets seal imdbgraph`
   from the repo root.
3. Commit both checked-in copies of the sealed secret that were updated by the
   command.
4. Push the change to `ci` so Flux can reconcile it.

The seal command now keeps `apps/imdbgraph/k8s/sealed-secret.yaml` and
`packages/infra/manifests/apps/stable/imdbgraph-sealed-secret.yaml` in sync so
resealing does not leave Flux with a stale copy.

For immediate recovery, applying the resealed manifest directly with
`kubectl apply` is acceptable, but Git still needs to be updated afterward
because Flux is Git-driven.

## Image publishing note

`az acr build` is currently not a reliable path for this repo's Dockerfile. The
Dockerfile uses BuildKit cache mounts during `pnpm dlx turbo prune`, and ACR
Tasks failed there during recovery.

Use local `docker buildx build --push` for now when you need to publish an image
for a fresh cluster.

## Local verification note

Shared Playwright config now writes transient test artifacts under the system
temp directory instead of `apps/*/test-results`, which avoids leaving root-owned
workspace artifacts behind after containerized runs.

## Fresh rebuild log

This is the exact sequence from a destroy-and-rebuild test of the real `staging`
environment.

### Destroy order

1. `pulumi destroy --stack staging --yes` in `apps/imdbgraph/infra`
2. `pulumi destroy --stack staging --yes` in `packages/infra`

### Provision order

1. `pulumi up --stack staging --yes` in `packages/infra`
2. `pulumi up --stack staging --yes` in `apps/imdbgraph/infra`
3. `az aks get-credentials --resource-group rg-aamini-staging --name aks-cluster-staging --overwrite-existing`
4. `docker buildx build --platform linux/amd64 --target production --build-arg APP_NAME=imdbgraph --build-arg PORT=3000 --build-arg NODE_VERSION=22 --tag aaministack.azurecr.io/imdbgraph:latest --push .`
5. `node --experimental-strip-types ./scripts/src/aamini.ts secrets seal imdbgraph`
6. `kubectl apply -f apps/imdbgraph/k8s/sealed-secret.yaml`

### Manual recovery actions that were required

- Cleared Flux custom resource finalizers manually so `flux-system` could finish
  deleting during `pulumi destroy`.
- Refreshed kubeconfig manually after the AKS cluster was recreated.
- Applied local manifests directly with `kubectl apply -k` because Flux could
  not reconcile from the current remote `ci` branch.
- Built and pushed the `imdbgraph` image manually because ACR did not already
  contain `aaministack.azurecr.io/imdbgraph:latest`.
- Resealed `imdbgraph` secrets manually because the new sealed-secrets
  controller key could not decrypt the checked-in secret.
- Granted `Network Contributor` manually to the AKS managed identity on
  `rg-aamini-staging` so Azure could attach the reserved public IP to the
  Traefik `Service`.
- Recreated the Traefik `Service` manually after the role assignment so Azure
  retried the load balancer binding.
- Updated the Cloudflare wildcard records manually during recovery while the
  static-IP binding issue was still unresolved.

### Gaps confirmed by the rebuild

#### 1. Flux namespace teardown is not clean

`pulumi destroy` stalled on the `flux-system` namespace because Flux custom
resources kept `finalizers.fluxcd.io` and the namespace stayed in `Terminating`.

Current impact:

- Platform destroy is not hands-off.
- Recovery may require manually patching Flux resources to remove finalizers.

#### 2. Cloudflare cannot safely manage the apex record here

The first fresh `pulumi up` failed when this stack tried to create the apex `@`
record. Cloudflare already had a root-domain CNAME, so the platform stack could
not take ownership of it.

Current impact:

- This stack should only manage the wildcard records it actually owns.
- Root-domain DNS ownership is still separate.

#### 3. AKS kubeconfig must be refreshed after cluster recreation

After the cluster was destroyed and recreated, local `kubectl` still pointed at
the dead API hostname from the old cluster.

Current impact:

- Any post-provision checks, reseal flows, or manual `kubectl apply` steps fail
  until `az aks get-credentials` is rerun.

#### 4. The remote `ci` branch does not contain the Flux source paths

During this rebuild, Flux reported errors like `kustomization path not found`
for `packages/infra/manifests/clusters/staging` and
`packages/infra/manifests/platform/crds` from the live `origin/ci` revision.

Current impact:

- The cluster cannot currently self-heal from Git alone.
- Recovery depended on direct `kubectl apply` from the local workspace.

#### 5. Sealed secrets still require resealing after cluster rebuild

The checked-in `SealedSecret` failed with `no key could decrypt secret` until it
was resealed for the new controller keypair.

Current impact:

- App startup is blocked until the runtime `Secret` is recreated.

#### 6. App image publication is still a manual prerequisite

The rebuilt cluster could not pull `aaministack.azurecr.io/imdbgraph:latest`
because the repository and tag did not exist in ACR yet.

Current impact:

- The app pod stays in `ImagePullBackOff` until someone builds and pushes the
  image.

#### 7. Static public IP binding needs a Pulumi-side RBAC fix

The Traefik `Service` could not attach the reserved public IP until the AKS
managed identity was granted `Network Contributor` on `rg-aamini-staging`.

Current impact:

- The origin IP from Pulumi can disagree with the actual live gateway IP.
- Cloudflare can end up pointing at a dead reserved IP until Azure RBAC is fixed
  and the `Service` is recreated.

This should be fixed in the Pulumi program rather than repeated manually.

### End state of this rebuild

- `packages/infra` and `apps/imdbgraph/infra` both reprovisioned successfully.
- The wildcard certificate is ready.
- The `imdbgraph` runtime `Secret` exists and the app pod is `Running`.
- The gateway now reports the reserved public IP `104.209.13.111`.
- Direct origin access to `https://imdbgraph.ariaamini.com` with
  `--resolve imdbgraph.ariaamini.com:443:104.209.13.111` returns `200`.
- Cloudflare was still returning `522` immediately after the final network fix,
  which looked like propagation or cache lag rather than an origin failure.
